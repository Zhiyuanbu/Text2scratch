1. Syntax Basics

These rules prevent most errors and make scripts readable.

One line = one command

Each line becomes one Scratch action in generated blocks.

move 10

Indent nested commands

Loop or if content must be indented below its parent.

repeat 10 move 10 end

Close with end

Every open control block needs a closing end.

if var(score) > 5 say "Great" end

Use quotes for sentence text

If text has spaces, wrap it in quotes.

say "Hello team"

2. Build Order

Use this sequence to avoid memory overload and broken scripts.

  1. Create shared data first: make_var, make_list, make_broadcast.
  2. Write stage behavior in stage_code = ... end.
  3. Add each sprite with sprite = "Name".
  4. Write sprite scripts in name_code = ... end.
  5. Run conversion and fix any Problems before export.

3. @ Expression Syntax

@ means "insert a value returned by another block".

Correct usage

set_var score @var(best_score)

Incorrect usage

@var(score)

Readable condition

if var(score) > 5 say "Great" end

4. Stage Coding Rules

Some Scratch blocks are sprite-only and cannot run on stage.

  • Put stage scripts inside stage_code = only.
  • If a line is stage-incompatible, move it into a sprite code section.
  • Use broadcasts to coordinate stage and sprites cleanly.

5. Multi-Sprite Template

Use this as your starter file for most projects.

make_var score 0 make_broadcast start_round stage_code = when_flag_clicked switch_backdrop_to backdrop1 broadcast start_round end sprite = "Cat" cat_code = when_broadcast_received start_round set_var score 0 repeat 5 move 10 change_var score 1 end end sprite = "Ball" ball_code = when_broadcast_received start_round if var(score) > 3 say "Ready" else think "Wait" end end

6. Extended Workflow Guide

Start every project by writing a short goal statement in plain language before touching commands, so your scripts map to one clear behavior.

Break that goal into tiny interactions such as movement, scoring, and scene transitions, then assign each interaction to stage logic or sprite logic.

Keep setup declarations together at the top of the file because variables, lists, and broadcasts are easiest to audit when they are not scattered across the project.

Use small scripts with obvious event names first, because long scripts make it harder to find where a state change began when something fails.

Whenever you add a new condition, test both the true path and the false path immediately so you never assume branch behavior from reading alone.

Use named broadcasts for milestones like round start, round end, or reset, and avoid generic names that become ambiguous later.

Prefer explicit variable resets at script start points so reruns do not carry stale values from a previous test cycle.

If a script behaves differently on stage and sprite targets, verify target compatibility first and move sprite-only actions into sprite code blocks early.

When using nested control blocks, align indentation carefully and close each group as soon as possible to reduce scope confusion.

After each successful milestone, save a known-good copy so future experiments can be rolled back quickly without rebuilding stable logic.

During final cleanup, shorten repetitive names, remove unused declarations, and keep only broadcasts and lists that are still part of runtime behavior.

Before export, run one complete end-to-end playthrough and record any warnings or odd transitions, then fix those while context is still fresh.
